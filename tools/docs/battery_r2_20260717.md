## Conclusions

1. **Portal ≈ fresh mode**: submit_013's portal ttft p50 = 66ms matches fresh-local 61ms, not shared-local 20ms
   → the real BTC prompts most likely have NO byte-stable turn-to-turn prefix (sliding-window suspicion confirmed-ish).
   TTFT is near its practical floor (full 4k prefill each turn); prefill-side levers are exhausted.
2. **Portal TPOT 4ms vs local 2.47ms** → the slice decodes ~1.6× slower (bandwidth), consistent with ~2ms
   theoretical fp8 + ~2ms/step overhead on the weak slice. cudagraph/async target exactly that overhead —
   neutral on the fast 4090 host but plausibly positive on the slice → worth one portal slot (submit_014).
3. **ngram spec decode is a clear loss** (TPOT 2.47→3.43ms, worse than bf16): drafting overhead > acceptance gains
   at this size/speed. Do NOT ship. (Caveat: synthetic random prompts depress acceptance; but a 40% TPOT
   regression is too big to be prompt-artifact.)
4. fp8 vs bf16: +11.9 (fresh) / +8.4 (shared) local ≈ +10.75 on portal (011→012). Calibration good.
5. chunk4096: no TTFT gain, slight shared-mode turn-1 regression (63 vs 41ms). Skip.
6. Remaining TPOT ideas beyond 014: suffix decoding via custom image (arctic-inference), draft-model spec decode
   (needs weights baked into image), deeper compilation tuning. GPQA still required for final-5 picks.
